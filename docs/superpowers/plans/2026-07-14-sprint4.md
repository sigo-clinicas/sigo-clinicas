# Sprint 4 — Diferencial e fechamento da Fase 1 (white-label, marketing, relatórios, convênios, hardening)

> **For agentic workers:** vertical slices, migration-first, cada slice em **um PR com CI verde** (squash `feat(s4-N): ...`). Testes obrigatórios por slice: **RLS (isolamento entre clínicas) + RBAC por papel + CÁLCULOS** (relatórios que somam dinheiro TÊM de bater com o financeiro do S3; segmentação de campanha e leitura de leads/depoimentos NÃO podem vazar entre clínicas). Portar do Base44 **citando a origem**; `service_role` só no servidor. Toda migration nova aplicada **também no remoto via MCP** (`xbnveoshbqtekctzwnur`) + `schema_migrations.version` normalizado; **se corrigir função no remoto, o arquivo de migration fica idêntico** (lição S3-3). **Auto-merge NÃO é automático no S4** — pedir autorização de merge quando o classificador de auto-mode exigir. Nada de escopo removido sem sinalizar.

**Goal:** Fechar a Fase 1 — white-label completo por vertical, marketing (Sala VIP, depoimentos+ranqueamento, campanhas com segmentação, destaque parametrizável), relatórios/dashboards **configuráveis** reconciliados com o financeiro, convênios (fechamento de guia por CSV — buffer) e o **hardening** de produção (CORS restrito, rate-limit, purga LGPD desligada, avisos de advisor) — e entregar o de-para consolidado dos Sprints 1–4.

**Arquitetura:** Next.js App Router + Supabase multi-tenant com RLS por `clinica_id`. Schema de marketing (cupom/campanha/depoimento/sala_vip/lead/lead_sala_vip) **já existe** desde S0; S4 constrói **Server Actions + UI** sobre ele. Agregações de relatório **no Postgres** (não puxar entidades ao browser). Operações públicas/privilegiadas via `service_role` server-side. Hardening endurece a superfície pública (CORS por origem, rate-limit por IP/janela) e a postura do banco (search_path, grants least-privilege, purga LGPD off-by-default).

**Tech Stack:** Next.js (RSC + Server Actions) · TS · Supabase (Postgres 15) · React 18 (`useState`/`useTransition`, sem TanStack/zod/Prisma) · shadcn/Radix · lucide-react · **gráficos self-contained (SVG/CSS)** — `recharts` NÃO está instalado; não adicionar · Vitest (`tests/rls` + `tests/unit`).

## Global Constraints

- **RLS por `clinica_id`** em toda tabela operacional; marketing usa `app.aplicar_padrao_tenant(..., array['proprietario','gerente'])` (select membro; escrita gestão). Actions gate por `exigirMarketing()`/papel + `.eq('clinica_id', sessao.clinicaAtual)`.
- **Destaque = admin-only** (`app.is_admin()`, RLS `clinica_destaque_admin`) → UI em `src/app/admin/*`, não `/painel`.
- **RPC pattern:** `security definer` + `set search_path=''`; autoriza `app.tem_papel(p_clinica_id, array[...])` → `42501`; violação → `23514`; `revoke execute from public` **e** `from anon` + `grant to authenticated` (RPC pública → `grant to service_role` só). `advisory_xact_lock` onde há corrida.
- **Contrato de retorno das Actions:** `{ erro: string | null; ok?: boolean }` (PT). Mapear `23505`/`23514`/`42501`/`23503`.
- **`clinica_id` sempre dos claims** (`getSessaoComClaims().clinicaAtual`), nunca do input.
- **Migrations:** `supabase/migrations/YYYYMMDDHHMMSS_slug.sql`, timestamp **acima de `20260714090800`** → S4 usa `20260714091000`+. Aplicar no remoto via MCP + normalizar + `npm run db:types` (helper `apply-types.js` no scratchpad).
- **Gates de CI (`qualidade` + `policies-rls`):** `lint` + `typecheck` + `test` + `test:rls` verdes. Feature sem policy RLS testada não fecha.
- **Reconciliação financeira (invariante do S3, teste de "done" dos relatórios):** para qualquer recorte, `Σ movimentacao_conta.valor (tipo='entrada') == Σ baixa_lancamento.valor (lançamentos tipo='receita')`; e no total `== Σ lancamento_financeiro.valor_pago (tipo='receita')`. Relatório de faturamento **recebido** usa `movimentacao_conta`/`baixa_lancamento` (regime de caixa), **nunca** `valor WHERE status='pago'` (bug do Base44). Comissão já entra como `despesa` via `apurar_comissao` — **não** somar `comissao.valor` de novo (dupla contagem).
- **Marketplace público** só expõe dado público (clínica `exibir_marketplace`, serviço `exibir_publico`, depoimento `aprovado+publicar_no_site`, sala_vip `aprovada+ativa`, cupom `ativo`) — testar não-vazamento em toda query pública nova.
- **Parametrizações pendentes da cliente (NÃO chumbar):** (1) modelo de cobrança do **destaque** (assinatura|período|leilão) — ligar a estrutura do S3-6, mas isolado em `nivel_destaque`/`marketplace_ranking_score`; (2) **KPIs** de relatório — conjunto default configurável em `clinica.config.dashboard`, não hard-coded; (3) comissão **por_evolucao vs por_venda** — suportar ambos por `clinica.config.base_comissao`. (4) **prazos de retenção LGPD** — purga desligada até a cliente definir.
- **Disparo real de WhatsApp = F2** (A5) — campanhas entregam **segmentação + público-alvo**; o botão de disparo é inerte/placeholder.

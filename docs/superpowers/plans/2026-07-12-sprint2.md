# Sprint 2 — Gestão do paciente (prontuário, anamnese pública, estoque-núcleo, LGPD)

> **For agentic workers:** vertical slices, migration-first, cada slice em PR com CI verde. Testes obrigatórios (RLS/RBAC/token) a cada slice. Portar do Base44 citando a origem. `service_role` só no servidor/Edge Function. Migrations novas aplicadas TAMBÉM no remoto via MCP (projeto `xbnveoshbqtekctzwnur`). Nada de escopo removido sem sinalizar.

**Goal:** Resolver a dor central da cliente ("agendei, atendi — cadê o prontuário/receituário/exames/antes-e-depois") com prontuário completo, anamnese configurável com preenchimento público por token, estoque-núcleo (dependência da rastreabilidade de insumos) e a camada de LGPD tratada como parte do fluxo.

**Arquitetura:** Next.js App Router + Supabase multi-tenant com RLS por `clinica_id`. CRUD autenticado via client da sessão (RLS decide); operações que cruzam a fronteira global↔tenant ou que quebram `insert().select()` viram RPC `SECURITY DEFINER` validando papel pelos claims; acesso público sem login (token) só via Edge Function Deno com `service_role` escopado ao registro do token. Fundação do prontuário/estoque já existe (migrations 0800/0900).

## Global Constraints

- RLS por `clinica_id` em toda tabela operacional (já garantido pela fundação).
- Fotos/áudio/documentos em buckets privados (`prontuario`/`documentos`), path `<clinica_id>/...`, nunca URL pública; exibição via signed URL server-side.
- `service_role` só em Edge Function/servidor; jamais no browser.
- Registro clínico é imutável/retido: sem hard-delete pelos papéis de escrita (retention-lock, S2-0).
- Decisões da cliente (07/12): entrada de estoque NÃO gera financeiro no S2 (→S3); odonto/mapa estética = componentes visuais no S2, persistência no S3; anamnese e avaliação SEPARADAS; prazos de retenção pendentes (landa colunas, sem purga).

- Geração de link: Server Action com client da sessão, `insert().select('token')`; **seta `expira_em = now() + clinica.validade_anamnese_dias`**. Link `${origin}/anamnese/${token}`.
- **Edge Function `anamnese-publica`** (`service_role` de Supabase secrets, nunca no client): `action:'get'` → valida token é uuid (senão 400) → `.eq('token',token).maybeSingle()` com JOIN embedded p/ nome do paciente + formulário; null→404; expirado→410; preenchido→already_filled. `action:'submit'` → **revalida obrigatórias no servidor** + `update ... .eq('token',token).eq('status','pendente')` (fecha duplo-submit). CORS restrito ao domínio do app; só POST. **Nenhuma query sem `.eq('token', ...)`.**
- Visualização autenticada: via client da sessão + RLS; paciente logado vê a própria (`resposta_anamnese_select_paciente`).
- RLS mantida SEM policy anon (defesa em profundidade).

**DoD (a slice NÃO fecha sem isto):** suíte de segurança do token — **isolamento por token** (clínica A não vaza em B), uuid inexistente→404, não-uuid→400 antes da query, expirado barrado, já-preenchido não sobrescreve, **duplo-submit concorrente** (2º afeta 0 linhas), **grep garantindo nenhum `select/update` em `resposta_anamnese` sem `.eq('token')`** na Edge Function, e **anon/authenticated key NÃO lê `resposta_anamnese`** (ausência de policy anon). E2E preencher via link sem login. Edge Function deployada no remoto (`supabase functions deploy` / MCP `deploy_edge_function`).

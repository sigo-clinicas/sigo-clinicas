## Global Constraints

- **RLS por `clinica_id` em toda tabela operacional** — já garantido pela fundação; qualquer coluna/tabela nova de S3 herda `aplicar_padrao_tenant` ou policy manual equivalente + teste.
- **Financeiro é restrito a `['proprietario','gerente']`** (leitura E escrita) — RLS já assim. `baixa_lancamento`/`movimentacao_conta` **não têm policy de INSERT/DELETE**: escrita só via RPC/`service_role`. Checar papel **também** na Server Action (defesa em profundidade).
- **Baixa/movimentação/venda/comissão nascem em RPC transacional.** Nunca INSERT direto de UI em `movimentacao_conta`/`baixa_lancamento`/`venda`/`pagamento`/`comissao`.
- **Saldo de conta** = view `saldo_conta_bancaria` (`saldo_inicial + Σ(entrada−saida)`); **nunca** coluna denormalizada.
- **RPC pattern (obrigatório):** `security definer` + `set search_path = ''`; todo ref schema-qualificado (`public.*`, `app.*`, `auth.*`); autoriza com `if not app.tem_papel(p_clinica_id, array[...]) then raise exception 'sem_permissao' using errcode='42501'; end if;`; violação de invariante/tenant → `raise ... using errcode='23514'`; `revoke execute ... from public;` **depois** `grant execute ... to authenticated;`. Isolamento de corrida via `pg_advisory_xact_lock(hashtextextended(...))`.
- **Contrato de retorno das Server Actions:** `{ erro: string | null; ok?: boolean; id?: string }` (chave PT `erro`). Mapear códigos PG: `23505`→já existe, `23514`→invariante/tenant, `42501`→sem permissão, `23503`→FK/não encontrado.
- **`clinica_id` vem SEMPRE dos claims** (`getSessaoComClaims().clinicaAtual`), nunca do input do cliente. Args de RPC são `p_`-prefixados e batem com a assinatura SQL.
- **Migrations:** `supabase/migrations/YYYYMMDDHHMMSS_slug.sql`, timestamp **acima de `20260712110300`** → S3 usa `20260714090xxx`. Cada arquivo abre com header `-- === S3-N ...`. Após aplicar: **MCP `apply_migration` em `xbnveoshbqtekctzwnur`** + `update supabase_migrations.schema_migrations set version = split_part(name,'_',1)` (normalização). `npm run db:types` regenera `src/lib/database.types.ts`.
- **Gates de "done" (CI: jobs `qualidade` + `policies-rls`):** `npm run lint` + `npm run typecheck` + `npm test` + `npm run test:rls` verdes. Uma feature sem policy RLS testada **não fecha**.
- **Marketplace público:** só expõe `clinica.exibir_marketplace` + `servico.exibir_publico` (+ `tabela_preco`/`item_tabela_preco`/`profissional`/`depoimento`/`cupom`/`sala_vip` públicos). **Dado de paciente/financeiro NUNCA vaza no público** (essas tabelas não têm policy anon). ## Decisões de schema desta fase (deltas) — tomadas, sinalizadas no de-para

O schema 0600/0700 é a base, mas o funil validado no Base44 exige 4 deltas pequenos. São **preservação de UX validada** (não remoção), registrados para a cliente poder objetar na homologação:

1. **Orçamento avulso (S3-1).** Base44 permite orçar sem paciente cadastrado (nome livre). `orcamento.paciente_id` é `NOT NULL`. **Delta:** tornar `paciente_id` **nullable**, adicionar snapshot `cliente_nome/cliente_telefone/cliente_email text` e `check (paciente_id is not null or cliente_nome is not null)`. Trigger `app.garantir_paciente_da_clinica` em `orcamento` **com `WHEN (new.paciente_id is not null)`** (herdado do S2-0). Caminho normalizado (paciente FK) continua primário.
2. **Item de produto de estoque no orçamento (S3-1).** `item_orcamento.servico_id` é `NOT NULL`; linhas de produto têm `item_estoque_id` e `servico_id` nulo. **Delta:** adicionar `item_estoque_id uuid references item_estoque(id) on delete set null`, `unidade text`; relaxar `servico_id` para nullable + `check ((servico_id is not null) <> (item_estoque_id is not null))` (exatamente um).
3. **Categorias hierárquicas (S3-3).** Base44 usa subcategorias + reordenação. **Delta:** `categoria_lancamento` += `pai_id uuid references categoria_lancamento(id) on delete set null`, `ordem int not null default 0`.
4. **Competência/origem da comissão (S3-5).** Para deduplicar apuração por execução e por mês. **Delta:** `comissao` += `competencia date`, `evolucao_sessao_id uuid references evolucao_sessao(id) on delete set null`; índices únicos parciais por origem (ver S3-5).

Deltas **descartados/adiados** (sinalizados): `regioes` continua `text[]` (lista plana de dentes/regiões — suficiente; odontograma estruturado = F2 se a cliente pedir); `profissional.exibir_publico` (opt-out por profissional) = **adiado** (hoje todo profissional ativo de clínica pública aparece — decisão da cliente); auto-expiração de orçamento por `validade_dias` = **adiado** (status `expirado` é manual, como no Base44).

- **RPC `public.vender_orcamento`** (`security definer`, `search_path=''`):
  ```
  vender_orcamento(
    p_clinica_id uuid, p_orcamento_id uuid,
    p_forma_pagamento public.forma_pagamento, p_data_venda date,
    p_parcelas jsonb   -- [{ "numero": int, "valor": numeric, "vencimento": "YYYY-MM-DD" }]

  ```
  Responsabilidades numa transação: (1) autoriza `['proprietario','gerente','recepcionista','assistente']`; (2) valida orçamento pertence à clínica e `status='aprovado'`; (3) `pg_advisory_xact_lock` no `orcamento_id`; (4) **valida `Σ parcelas.valor = orcamento.valor_final`** (tolerância `0.01`, senão `23514`); (5) insere `venda` (a UNIQUE em `orcamento_id` garante 1 venda por orçamento — 2ª tentativa → `23505`); (6) para cada parcela insere `lancamento_financeiro` (`tipo='receita'`, `valor`, `data_vencimento=vencimento`, `status='pendente'`, `venda_id`, `paciente_id` do orçamento, `categoria_id` = categoria "Vendas/Serviços" da clínica se existir, senão null) e um `pagamento` (`venda_id, numero_parcela, valor, vencimento, forma_pagamento, lancamento_id`=lançamento recém-criado). Retorna `venda_id`. `revoke from public` + `grant to authenticated`.
- **Server Action `venderOrcamento`:** calcula o array de parcelas no servidor a partir de `valor_final` e `n` (parcelas iguais arredondadas a 2 casas; **última parcela absorve o resto** para somar exato), vencimentos mensais a partir de `data_venda`. Chama a RPC. Mapeia `23505`→"Orçamento já vendido", `23514`→"Parcelas não somam o total".
- **Tab Vendas & Faturamento:** query `venda` join `pagamento` (+ `orcamento.valor_final`). KPIs: `totalFaturado = Σ valor_final`, `totalRecebido = Σ pagamento.valor where pago`, `totalPendente = faturado − recebido`. Marcar parcela recebida **não** é INSERT de UI aqui — é a baixa do `lancamento_financeiro` vinculado (S3-3); a tab mostra status derivado do lançamento.

**Interfaces (produz):** `venda` + `pagamento[]` + `lancamento_financeiro` (receita/pendente) vinculados → consumidos pelo financeiro (S3-3). RPC `vender_orcamento`.

**DoD:**
